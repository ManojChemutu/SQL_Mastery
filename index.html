```
LESSON 1.1

Your First SQL Query

Writing and understanding a complete SQL statement from scratch

Beginner

Every SQL query follows a logical pattern. Let's break down the anatomy of a SELECT statement — the most common SQL operation you'll write.

SQL
basic_select.sql
``` 
-- Basic SELECT: retrieve specific columns from a table
SELECT
    employee_id,         -- column 1
    first_name,          -- column 2
    last_name,           -- column 3
    salary               -- column 4
FROM
    employees            -- the source table
WHERE
    salary > 50000       -- filter condition
ORDER BY
    salary DESC          -- sort results
LIMIT 10;                -- return top 10 rows
```
💡 Order matters! SQL clauses execute in a specific logical order: FROM → WHERE → GROUP BY → HAVING → SELECT → ORDER BY → LIMIT

SQL Clause Summary

  • SELECT — specifies which columns to return
  • FROM — names the table(s) to query
  • WHERE — filters rows based on conditions
  • ORDER BY — sorts results (ASC or DESC)
  • LIMIT / FETCH FIRST — restricts how many rows return
🚀 Pro Tip: Use SELECT * only during exploration. In production, always specify column names — it's faster, clearer, and breaks fewer things when schemas change.
```

```
Primary Keys vs Foreign Keys

SQL
keys.sql
``` 
-- Primary Key: uniquely identifies each row
CREATE TABLE customers (
    customer_id  INT          PRIMARY KEY,
    first_name   VARCHAR(50) NOT NULL,
    email        VARCHAR(100) UNIQUE NOT NULL,
    pan_number   CHAR(10)    UNIQUE
);

-- Foreign Key: links to another table's PK
CREATE TABLE accounts (
    account_id   INT             PRIMARY KEY,
    customer_id  INT             NOT NULL,
    balance      DECIMAL(15,2) DEFAULT 0.00,
    account_type VARCHAR(20)   CHECK (account_type IN ('SAVINGS', 'CURRENT', 'NRI')),
    FOREIGN KEY (customer_id) REFERENCES customers(customer_id)
        ON DELETE CASCADE
        ON UPDATE CASCADE
);
```

Normalization — Eliminating Redundancy

1NF — First Normal Form
Each column holds atomic (indivisible) values. No repeating groups. Each row is unique.
2NF — Second Normal Form
Must be in 1NF. Every non-key attribute is fully dependent on the entire primary key (eliminates partial dependencies).
3NF — Third Normal Form
Must be in 2NF. No transitive dependencies — non-key columns shouldn't depend on other non-key columns.
```

```
SQL
join_examples.sql
``` 
-- INNER JOIN: customers with accounts (only matched records)
SELECT
    c.first_name,
    c.last_name,
    a.account_id,
    a.balance,
    a.account_type
FROM customers c
INNER JOIN accounts a ON c.customer_id = a.customer_id
WHERE a.status = 'ACTIVE';

-- LEFT JOIN: ALL customers, even those with no account
SELECT
    c.customer_id,
    c.first_name,
    COALESCE(a.account_id, 'No Account') AS account,
    a.balance
FROM customers c
LEFT JOIN accounts a ON c.customer_id = a.customer_id;

-- Multi-table JOIN: customers + accounts + transactions
SELECT
    c.first_name,
    a.account_type,
    t.amount,
    t.txn_type,
    t.txn_date
FROM customers c
INNER JOIN accounts     a  ON c.customer_id  = a.customer_id
INNER JOIN transactions t  ON a.account_id   = t.from_account
WHERE t.txn_date >= DATE_SUB(NOW(), INTERVAL 30 DAY)
ORDER BY t.txn_date DESC;

-- SELF JOIN: find employees who report to the same manager
SELECT
    e1.first_name AS employee,
    e2.first_name AS manager
FROM employees e1
LEFT JOIN employees e2 ON e1.manager_id = e2.employee_id;
```
🚀 Performance Tip: Always JOIN on indexed columns. Joining on unindexed foreign keys causes full table scans — devastating at scale in BFSI with millions of transactions.
```

LESSON 6.1

Aggregate Functions & GROUP BY

From raw rows to meaningful summaries

SQL
aggregations.sql
-- Basic aggregate functions
SELECT
    COUNT(*)                    AS total_transactions,
    COUNT(DISTINCT from_account) AS unique_accounts,
    SUM(amount)                  AS total_volume,
    AVG(amount)                  AS avg_transaction,
    MIN(amount)                  AS min_txn,
    MAX(amount)                  AS max_txn
FROM transactions
WHERE status = 'SUCCESS'
  AND txn_date >= '2024-01-01';

– GROUP BY: aggregate per category
SELECT
txn_type,
COUNT(*)          AS count,
SUM(amount)       AS total_amount,
ROUND(AVG(amount), 2) AS avg_amount
FROM transactions
GROUP BY txn_type
HAVING COUNT(*) > 100    – HAVING filters AFTER grouping (not WHERE!)
ORDER BY total_amount DESC;

– Monthly transaction summary — useful for dashboards
SELECT
DATE_FORMAT(txn_date, ’%Y-%m’) AS month,
COUNT(*)                        AS txn_count,
SUM(amount)                     AS monthly_volume,
COUNT(DISTINCT from_account)   AS active_accounts
FROM transactions
WHERE status = ‘SUCCESS’
GROUP BY DATE_FORMAT(txn_date, ’%Y-%m’)
ORDER BY month;
```
💡 WHERE vs HAVING: WHERE filters individual rows BEFORE grouping. HAVING filters groups AFTER GROUP BY is applied. You cannot use aggregate functions in WHERE.
```

```
CTEs — Common Table Expressions (WITH clause)

SQL
cte_examples.sql
``` 
-- CTE: readable alternative to nested subqueries
WITH monthly_volumes AS (
    SELECT
        DATE_FORMAT(txn_date, '%Y-%m') AS month,
        SUM(amount) AS total
    FROM transactions
    WHERE status = 'SUCCESS'
    GROUP BY DATE_FORMAT(txn_date, '%Y-%m')
),
avg_monthly AS (
    SELECT AVG(total) AS avg_volume
    FROM monthly_volumes
)
SELECT
    mv.month,
    mv.total,
    am.avg_volume,
    ROUND((mv.total - am.avg_volume) / am.avg_volume * 100, 1) AS pct_vs_avg
FROM monthly_volumes mv, avg_monthly am
ORDER BY mv.month;

-- Recursive CTE: organizational hierarchy
WITH RECURSIVE org_chart AS (
    -- Anchor: top-level employees (no manager)
    SELECT employee_id, first_name, manager_id, 0 AS level
    FROM employees
    WHERE manager_id IS NULL

    UNION ALL

    -- Recursive: employees reporting to found employees
    SELECT e.employee_id, e.first_name, e.manager_id, oc.level + 1
    FROM employees e
    JOIN org_chart oc ON e.manager_id = oc.employee_id
)
SELECT * FROM org_chart ORDER BY level;

LESSON 8.1

OVER() Clause — The Window Concept

Performing calculations across related rows without collapsing the result set

Advanced

Window functions operate on a "window" of rows related to each current row — unlike GROUP BY which collapses rows into groups. The key: they return a value for every row.

```
SQL
window_functions.sql
``` 
-- Ranking Functions
SELECT
    employee_id,
    department,
    salary,
    ROW_NUMBER()   OVER (PARTITION BY department ORDER BY salary DESC) AS row_num,
    RANK()         OVER (PARTITION BY department ORDER BY salary DESC) AS rank,
    DENSE_RANK()   OVER (PARTITION BY department ORDER BY salary DESC) AS dense_rank,
    PERCENT_RANK() OVER (PARTITION BY department ORDER BY salary DESC) AS pct_rank,
    NTILE(4)       OVER (PARTITION BY department ORDER BY salary DESC) AS quartile
FROM employees;

-- LAG / LEAD: compare with previous/next row
SELECT
    account_id,
    txn_date,
    amount,
    LAG(amount, 1) OVER (PARTITION BY account_id ORDER BY txn_date) AS prev_amount,
    LEAD(amount, 1) OVER (PARTITION BY account_id ORDER BY txn_date) AS next_amount,
    amount - LAG(amount, 1) OVER (PARTITION BY account_id ORDER BY txn_date)
        AS change_in_amount
FROM transactions;

-- Running totals (cumulative sum)
SELECT
    txn_date,
    amount,
    SUM(amount) OVER (
        PARTITION BY account_id
        ORDER BY txn_date
        ROWS BETWEEN UNBOUNDED PRECEDING AND CURRENT ROW
    ) AS running_balance,
    AVG(amount) OVER (
        PARTITION BY account_id
        ORDER BY txn_date
        ROWS BETWEEN 6 PRECEDING AND CURRENT ROW
    ) AS rolling_7day_avg
FROM transactions
WHERE account_id = 1042;
```
🚀 RANK vs DENSE_RANK: If two employees share rank 2, RANK skips to 4 next. DENSE_RANK goes to 3. Use DENSE_RANK for leaderboards where you don't want gaps.
```

LESSON 9.1

How Indexes Work & When to Use Them

B-tree indexes, composite indexes, and the cost of over-indexing

Advanced

An index is a separate data structure (usually a B-tree) that holds sorted values of a column alongside pointers to the actual rows. Instead of scanning every row, the database jumps directly to the matching rows.

```
SQL
indexes.sql
``` 
-- Create a simple index
CREATE INDEX idx_txn_date
    ON transactions(txn_date);

-- Composite index: column order matters!
-- Efficient for: WHERE status = 'SUCCESS' AND txn_date > ...
-- NOT efficient for: WHERE txn_date > ... (alone)
CREATE INDEX idx_status_date
    ON transactions(status, txn_date);

-- Unique index: enforces uniqueness + fast lookup
CREATE UNIQUE INDEX idx_pan_unique
    ON customers(pan_number);

-- Partial index (PostgreSQL): only index a subset
CREATE INDEX idx_active_accounts
    ON accounts(customer_id)
    WHERE status = 'ACTIVE';

-- EXPLAIN: see how the database executes your query
EXPLAIN ANALYZE
SELECT * FROM transactions
WHERE status = 'SUCCESS'
  AND txn_date >= '2024-01-01';

-- List all indexes on a table (MySQL)
SHOW INDEXES FROM transactions;

-- Drop an index
DROP INDEX idx_txn_date ON transactions;
```

Index Anti-Patterns — What Kills Performance

SQL
antipatterns.sql
``` 
-- ❌ Function on indexed column = index ignored
SELECT * FROM transactions
WHERE YEAR(txn_date) = 2024;  -- full scan!

-- ✅ Fix: use range instead
SELECT * FROM transactions
WHERE txn_date BETWEEN '2024-01-01' AND '2024-12-31';

-- ❌ Implicit type conversion kills indexes
WHERE account_id = '12345'; -- account_id is INT, casting to string

-- ❌ Leading wildcard = full scan
WHERE name LIKE '%Kumar';   -- can't use B-tree index

-- ✅ Trailing wildcard uses index
WHERE name LIKE 'Kumar%';   -- uses index efficiently

LESSON 10.1

ACID Properties & Transaction Control

COMMIT, ROLLBACK, SAVEPOINT, and isolation levels

Advanced
⚛️
Atomicity
All-or-nothing. If any part of a transaction fails, the entire transaction is rolled back.
✅
Consistency
A transaction can only bring the database from one valid state to another, maintaining all constraints.
🔒
Isolation
Concurrent transactions execute as if they were serial. One transaction can't see another's uncommitted changes.
💾
Durability
Once committed, changes are permanently saved, even if the system crashes immediately after.
```
SQL
transactions.sql
``` 
-- Fund transfer: ACID-safe transaction
START TRANSACTION;

SAVEPOINT before_transfer;

-- Debit sender
UPDATE accounts
SET balance = balance - 10000
WHERE account_id = 1001
  AND balance >= 10000;  -- prevent overdraft

-- Check if update happened (row affected)
IF ROW_COUNT() = 0 THEN
    ROLLBACK TO before_transfer;
    -- signal: insufficient funds
END IF;

-- Credit receiver
UPDATE accounts
SET balance = balance + 10000
WHERE account_id = 2002;

-- Log the transaction
INSERT INTO transactions
    (from_account, to_account, amount, txn_type, status)
VALUES
    (1001, 2002, 10000, 'NEFT', 'SUCCESS');

COMMIT;  -- make permanent

-- Isolation Levels (least → most strict)
SET TRANSACTION ISOLATION LEVEL READ UNCOMMITTED;  -- dirty reads possible
SET TRANSACTION ISOLATION LEVEL READ COMMITTED;    -- most BFSI systems
SET TRANSACTION ISOLATION LEVEL REPEATABLE READ;   -- MySQL default
SET TRANSACTION ISOLATION LEVEL SERIALIZABLE;      -- strictest
```

Case 1: Fraud Detection — Unusual Transaction Patterns

Citi Risk Team · AML & Fraud Analytics

Advanced

Detect customers who made more than 5 transactions within 10 minutes — a common fraud pattern (card testing, credential stuffing).

SQL
fraud_detection.sql
``` 
WITH rapid_txns AS (
    SELECT
        from_account,
        txn_date,
        amount,
        -- Count transactions within 10-minute window
        COUNT(*) OVER (
            PARTITION BY from_account
            ORDER BY txn_date
            RANGE BETWEEN INTERVAL '10' MINUTE PRECEDING AND CURRENT ROW
        ) AS txns_in_10min,
        SUM(amount) OVER (
            PARTITION BY from_account
            ORDER BY txn_date
            RANGE BETWEEN INTERVAL '10' MINUTE PRECEDING AND CURRENT ROW
        ) AS amount_in_10min
    FROM transactions
    WHERE status = 'SUCCESS'
      AND txn_date >= DATE_SUB(NOW(), INTERVAL 24 HOUR)
),
flagged_accounts AS (
    SELECT DISTINCT from_account
    FROM rapid_txns
    WHERE txns_in_10min >= 5
       OR amount_in_10min > 100000
)
SELECT
    c.first_name,
    c.last_name,
    c.email,
    a.account_id,
    a.balance,
    'FRAUD_ALERT' AS risk_flag
FROM flagged_accounts fa
JOIN accounts  a ON fa.from_account  = a.account_id
JOIN customers c ON a.customer_id    = c.customer_id;

        SUM(t.amount)                    AS txn_volume_ytd

    JOIN accounts     a ON c.customer_id = a.customer_id
    LEFT JOIN transactions t ON a.account_id = t.from_account
        AND YEAR(t.txn_date) = YEAR(CURDATE())
    GROUP BY c.customer_id, c.first_name, c.last_name
)
SELECT
    customer_name,
    account_count,
    FORMAT(total_balance, 2)       AS balance,
    txn_count_ytd                   AS transactions,
    FORMAT(txn_volume_ytd, 2)       AS volume,
    CASE
        WHEN total_balance > 1000000 THEN '💎 Premium'
        WHEN total_balance > 100000  THEN '⭐ Standard'
        ELSE                              '📌 Basic'
    END AS tier,
    RANK() OVER (ORDER BY total_balance DESC) AS wealth_rank
FROM customer_metrics
ORDER BY total_balance DESC
LIMIT 100;

```
AdvancedQ69. What is MVCC and how does Oracle implement it?▼

Multi-Version Concurrency Control (MVCC) allows readers and writers not to block each other. Oracle stores old versions of data in the UNDO tablespace. Readers always see a consistent snapshot as of their query's SCN (System Change Number). Writers get row-level locks; readers never wait for writers.

💡 BFSI: This enables real-time balance reporting while batch jobs update account records simultaneously.

AdvancedQ70. ORA-01555 Snapshot Too Old — cause and fix?▼

Occurs when a long-running query needs an old version of a block from UNDO, but that UNDO has been overwritten. Causes: UNDO retention too low, UNDO tablespace too small, or very long transactions.

Fix: Increase UNDO_RETENTION parameter, increase UNDO tablespace size, use GUARANTEE retention, or break long-running queries into smaller batches.
```

```
AdvancedQ95. ACID guarantees in Oracle — how is Durability ensured?▼

Oracle ensures ACID through:

  • Atomicity: UNDO segments — failed transactions are automatically rolled back
  • Consistency: Constraints, triggers, MVCC
  • Isolation: Row-level locking + MVCC (no dirty reads)
  • Durability: REDO log — every change is written to redo log before confirming commit. Even if DB crashes, redo allows recovery. Log Writer (LGWR) writes redo synchronously on COMMIT.

ExpertQ98. CAP theorem — implications for distributed SQL in banking?▼

CAP theorem: a distributed system can guarantee only 2 of 3 — Consistency, Availability, Partition tolerance. Since network partitions are unavoidable, systems choose:

  • CP systems (Oracle RAC, traditional RDBMS): Strong consistency, may be unavailable during partition — appropriate for payments and ledger
  • AP systems (Cassandra, DynamoDB): Always available, eventually consistent — appropriate for fraud score caching, customer preference data

💡 BFSI: Financial systems must be CP for monetary transactions. AP systems are used for supporting data (notifications, recommendations, analytics). Never use AP for core ledger.

ExpertQ99. Zero-downtime schema migration in production?▼

The Expand-Contract pattern:

  1. Expand: Add new column as nullable (ALTER TABLE ADD — instant in Oracle for NULL cols). Deploy app that writes to both old and new columns.
  2. Migrate: Backfill new column in batches (UPDATE 1000 rows at a time with COMMIT — no single large lock)
  3. Switch: Update app to read from new column only
  4. Contract: Drop old column after confirming no references

For index additions: CREATE INDEX ... ONLINE (no table lock). For large table restructuring: partition exchange — swap a rebuilt partition in under a second.
```
